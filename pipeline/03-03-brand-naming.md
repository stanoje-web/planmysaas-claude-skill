# Stage 3.3 — Brand & naming

Generate `./planmysaas-blueprint/03-03-brand-naming.md`. Read `01-01-problem-validation.md` (for any existing project name and its anti-goals) and `03-01-competitive-landscape.md` (for the names and positioning to differentiate against) first.

## Output structure

```markdown
# Brand & naming — <Project name>

## Existing name evaluation

If `01-01-problem-validation.md` already has a project name, score it against 4 criteria, each with a 1-line justification:
- **Memorability** — does it stick after one hearing?
- **Pronunciation** — can it be said out loud unambiguously, read from a slide, spelled from audio?
- **Category fit** — does it signal the right space without being generic?
- **Differentiation** — does it stand apart from every competitor named in `03-01-competitive-landscape.md`?

End with a verdict: **Keep / Refine / Replace**, plus 1 sentence on why.

If no name exists yet, state that plainly and skip straight to candidates.

## Name candidates (5-8)

For each candidate: name · naming pattern used · one-line rationale tying it back to the problem or persona.

Patterns:
- **Descriptive** — states what it does
- **Compound** — two real words fused
- **Invented / coined** — new word, no dictionary meaning
- **Metaphor / analogy** — borrows meaning from an unrelated concept
- **Founder-name-based** — built on a person's name

Cover at least 3 of the 5 patterns across the candidate set — don't submit 6 descriptive names and call it done.

## Tagline options (3-5)

Short taglines, each pairing a candidate name (or the existing name) with a distinct angle — problem-led, outcome-led, audience-led, category-led.

## Domain & handle availability

Reminder only: domain and social handle availability must be checked manually by the founder before committing to any name. Do not assume, imply, or state that any specific domain or handle is available or taken — this section names the check, it never performs it.

## Brand voice guidelines

3-4 voice adjectives (e.g. "direct, technical, a little irreverent"), each with a 1-sentence definition of what it means in practice for this product, plus one example sentence written in that voice — an empty-state message, an onboarding line, or an error message specific to this product.
```

## Quality rules

- Never claim a domain, trademark, or social handle is available or unavailable — always defer that check explicitly to the founder. This applies to every candidate name, including any existing one.
- Candidate names must be differentiated from every competitor named in `03-01-competitive-landscape.md` — flag it explicitly if a candidate sounds too close to an existing player's name or tagline.
- No placeholder names like "Project X" or "YourApp" — every candidate must be a real, usable word or coinage a founder could plausibly register today.
- The example voice sentence must be product-specific copy, not a generic tagline restated in different words.
- If the founder's anti-goals (from `01-01-problem-validation.md`) rule out a tone or category, candidates must respect that — don't propose a playful name for a product whose anti-goals demand "no gimmicks."

## Tone

Analytical. Data-shaped. Read like a McKinsey one-pager, not a blog post.
