# Stage 2.3 — Demand signals

Generate `./planmysaas-blueprint/02-03-demand-signals.md`. Read `01-01-problem-validation.md`, `01-02-customer-personas.md`, and `01-03-jobs-to-be-done.md` first — every signal below must point back to the same audience and job, not a generic search for the category.

## Output structure

```markdown
# Demand signals — <Project name>

## How to use this file

One sentence: everything below is checkable by the founder in under a day, with no paid tools required.

## Search-intent signals

5-8 keyword phrases the target persona would actually type, each with:
- Phrase
- Intent type (informational / comparison / transactional)
- Directional volume label (High / Medium / Low — not a fabricated number)
- What finding real volume here would confirm

## Community signals

3-5 named places the audience actually congregates (specific subreddits, forums, Facebook/Discord/Slack groups, niche communities — named, not generic "social media"). For each:
- Community name + where it lives
- What complaint pattern to search for there (a phrase or thread topic to look up)
- What a strong signal looks like (e.g. "recurring monthly thread" vs "one post two years ago")

## Competitor review-mining check

Name 2-3 adjacent or substitute tools to check on G2 / Capterra / app store reviews. For each: what 1-2 star complaint pattern would validate this gap if found, and where exactly to look (product name + review platform).

## Pricing-signal check

Is this audience already paying for something adjacent to this job? Name a specific existing paid product/service they plausibly pay for today (or state plainly if none exists). State what price point that sets as an anchor and what it implies about willingness to pay for this idea.

## Demand-signal checklist (8-10 items)

A checklist of concrete, individually verifiable claims. Each item is phrased as something to go check, with a placeholder rating column the founder fills in after investigating:

| # | Signal to check | How to check it | Rating (Yes/Weak/No) |
|---|---|---|---|
| 1 | ... | ... | *(founder fills in)* |

## Overall demand verdict

One paragraph: given what's checkable (not yet checked), is this a Strong / Moderate / Weak demand case going in, and what's the single check that would move the needle most if it comes back positive.
```

## Quality rules

- Every signal must be something the founder can go verify in under a day with a browser — no paid research panels, no fabricated survey data, no invented statistics.
- Volume labels must stay directional (High/Medium/Low) — never state a specific search volume number as fact.
- Community names must be real and specific enough to visit (an actual subreddit name, an actual named Discord/Slack community if known, or "search for X-named communities on Y platform" if the exact one isn't known) — reject "online forums" as a signal source.
- The pricing-signal check must name a real adjacent product/service or explicitly say none exists — don't imply willingness to pay without an anchor.
- The checklist ratings column stays blank (Yes/Weak/No) — this file proposes what to check, it does not fabricate the results.

## Tone

Practical. Written like a field-research checklist for a founder heading out to do 3 hours of homework, not a market report.
